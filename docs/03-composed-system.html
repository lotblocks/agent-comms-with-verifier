Reference architecture · No. 03

The composed system.

Two primitives, one substrate. The message bus carries the conversation between agents; the verifier sits inside each worker, gating every reply. Together they're a working model for trustworthy multi-agent systems.

READ 12 min STATUS Working code, demo green UPDATED 2026·05·18 SERIES Ref. 01 + 02 → 03

Reference No. 01 designed the bus — how agents address each other and exchange messages without a conductor. Reference No. 02 designed the verifier — how a worker's output gets adversarially reviewed before it ships. This piece is what happens when you put them together, and what you should actually do next.

§ 01 — RecommendationWhat to do, in three moves

The opinionated answer

Treat them as two independent skills that compose

Move 1: save the existing verifier skill draft. Then run an update to add the LLM backend and declare ANTHROPIC_API_KEY as a credential. One follow-up turn; nothing exotic.

Move 2: save the bus and the two demo agents as a second skill — agent-bus. The bus is genuinely reusable across many setups; it shouldn't be a private implementation detail of the verifier.

Move 3: use them together by convention, not by glue code. Every worker agent calls the verifier on its own output before publishing. The bus carries the verification record as part of the response payload. Receivers read verification.status to decide whether to trust the answer.

That's the entire integration. Not a third skill, not a coupling layer, not a framework — just two clean primitives and one discipline.

§ 02 — System architectureWhere each piece sits

Here's the layout of the composed system. The bus is the only shared dependency. Each agent has its own runtime (its own process, its own subscriptions, its own credentials). The verifier is private to each worker — different agents can use different verifier configurations.

Two agents on a shared bus. Beta has the verifier privately inside its process — Alpha doesn't see it directly, but it sees the verification record that travels with every reply. The blackboard and Slack edge are optional additions for state and human escalation.

The important property of this layout: nothing is coupled. The bus knows nothing about verification. The verifier knows nothing about the bus. Beta is the only piece that knows about both, and it knows because it does both — it receives messages and it produces verified work. New worker agents follow the same pattern: subscribe to your inbox, run incoming requests through the verifier, reply with the verification record attached.

§ 03 — Message flowOne full request, end to end

Below is what actually happens when Alpha asks Beta to do something. Each lifeline is a process or component; each horizontal arrow is a message or function call. Time flows top to bottom.

Time flows top to bottom. The verifier loop happens entirely inside Beta's process — the bus never sees the remediation. Alpha receives one response message that already encodes the work, the attempts taken, and what the verifier found.

§ 04 — WalkthroughWhat happens step by step

Following the flow chart above. Each step is one box's worth of work; together they're a complete agent-to-agent interaction with built-in trust.

1
Alpha → Bus

Alpha sends a directly-addressed request

Alpha calls bus.send_direct(to=beta, payload={task, intent, skill_doc, max_attempts}). A new conversation_id is minted; the envelope carries hop_count=0, a fresh UUID, and a 5-minute TTL. The bus inserts a row and the message becomes visible in Beta's inbox.

2
Beta ← Bus

Beta picks up the request from its inbox

Beta polls bus.receive(agent_id=beta, subscriptions=["inbox.beta"]). The bus atomically claims any unclaimed messages addressed to Beta — atomic so that if multiple Beta replicas were running, only one would get the work. TTL-expired messages are swept on the same read.

3
Beta → Verifier orchestrator

Beta hands the work to the verifier wrapper

Instead of running its builder script directly, Beta invokes RunSkillVerified with the builder path, the intent from Alpha's request, and a verifier configuration. The orchestrator owns the rest of the loop. Beta has now delegated both the work and the self-check in one call.

4
Verifier → Builder (attempt 1)

The orchestrator runs the builder for the first time

The builder is a subprocess. No remediation prompt yet. In the demo, the toy builder returns an output deliberately missing two fields (no amount, no timestamp) so we exercise the failure path.

5
Verifier (internal)

Decomposition into atomic claims, verdicts assigned

The verifier produces 5 claims: existential ("output is non-empty"), structural ("output is JSON"), semantic ("addresses the user's intent"), factual ("output is timestamped from today"), negative ("no credentials leaked"). On the missing-fields output, the semantic claim fails and the factual claim is unverifiable. Three pass.

6
Verifier → Builder (attempt 2)

Remediation prompt built; builder re-invoked

The orchestrator builds a structured remediation prompt naming each failed claim with reasoning and what the verifier needs. It re-runs the builder with REMEDIATION_PROMPT in the environment. The builder reads it and produces a complete output. The verifier re-decomposes and gets all-pass.

7
Verifier → Beta

Verification record produced and returned

The orchestrator returns a single JSON record to Beta: { result, attempts:2, verification: { status:"verified", claims:[...], gap_report:{...} } }. The session-wide gap report still surfaces the originally-unverifiable claim, so even on success, the doc-improvement signal is preserved.

8
Beta → Bus → Alpha

Response sent back over the bus

Beta packages the verifier's record as the response payload and calls bus.send_direct(to=alpha, payload=..., conversation_id=same, reply_to=msg_id, msg_type="response"). The conversation thread is preserved by id.

9
Alpha (trust check)

Alpha decides whether to accept

Alpha reads verification.status. On verified, accept. On partial, accept but note the gap report. On failed, refuse — escalate to a human channel, retry with a different worker, or fall back to a known-good path. The decision is the agent's, not the verifier's.

The trust handoff is the entire point. Alpha doesn't have to take Beta's word for it. Alpha can see the claims, the evidence, the confidence, the cost, the model that did the verifying. That's what makes the agents work together rather than just exchange data.

§ 05 — Pros of this setupWhy these two primitives belong together

Plenty of multi-agent frameworks exist. This one is small, but it has properties that matter more than feature count.

PRO 01

Composition without coupling

The bus knows nothing about verification. The verifier knows nothing about the bus. They compose because Beta's response payload happens to carry a verification record — that's all. Either piece works without the other.

PRO 02

Trust travels with the message

Receivers don't have to assume the sender did their homework. Every reply carries the verifier's record — status, claims, evidence, confidence, cost. The receiver can apply its own acceptance policy.

PRO 03

Self-correction inside workers

Workers fix their own output before publishing. A single remediation attempt is often all it takes. The user does not have to discover the failure later — it never leaves the worker.

PRO 04

Independent failure modes

The verifier runs on a different model family than the builder. The bus is a separate process from any agent. The blackboard is separate from messages. Same-cause failures across layers are unlikely; one broken piece doesn't take everything down.

PRO 05

Documentation flywheel

Every unverifiable claim becomes a candidate doc improvement, surfaced via the existing draft-card flow. Skills get better not because someone rewrote them, but because verification kept surfacing what they failed to say.

PRO 06

Bounded blast radius

TTL on every envelope. hop_count hard cap. max_attempts on every verification. Per-attempt budget cap on verifier cost. Heartbeat-based agent staleness. Loops cannot persist; runaway costs cannot accumulate.

PRO 07

Fully observable

Every message is a row in SQLite. Every claim is recorded with its evidence. Every conversation can be replayed. You don't need a separate logging system to debug what happened — the bus is the log.

PRO 08

Production swap-in is a single change per layer

SQLite → Postgres LISTEN/NOTIFY or Redis pub-sub for the bus. Mock → LLM verifier (same verify() signature). Subprocess builder → real Hyperagent skill via RunWithCredentials. Each swap is local; nothing else has to change.

§ 06 — What this isn'tHonest limitations

The system above is small and complete, but it isn't everything. Here's what it deliberately doesn't do, so you know what to add later vs. what's a non-goal.

Real-time low-latency coordination

SQLite + polling is fine for the conversational pace of agents discussing work (seconds to minutes). For sub-100ms coordination, swap to NATS or Redis Streams. The agent code does not change.

Agentic verifier with its own tools

The current LLM verifier is single-call — it inspects the builder's output text and reasons. Behavioral claims that require running code are marked unverifiable. An agentic verifier that could make tool calls is a v2 upgrade with the same external signature.

Chain-of-skill verification

The verifier wraps a single builder. If a high-level task chains three skills, the current pattern verifies each call independently. Cross-skill claims (e.g. "the data fetched in step 1 is what step 2 acted on") are deferred.

Human-in-the-loop attestation

Every claim is decided automatically. For high-stakes operations (large money movement, deletions, public posts), a human_attestation claim type is a natural extension — fires when present, routes to a human channel for approval before the worker can publish.

Cross-tenant isolation

The current bus assumes all agents trust the same SQLite file. Multi-tenant deployments need separate buses per tenant or per-row ACLs. Not difficult, just not done.

§ 07 — What's nextThree concrete moves

Three things would make this real, in order:

1

Save both as skills, swap the verifier to LLM mode

Save the existing verifier draft, then update it to add the LLM backend and declare the credential. Create a sibling agent-bus skill from the working code in /agent/workspace/demos/comms-with-verifier/. With both saved, any future agent can FetchSkillScripts and use them.

2

Build the observability surface

A static HTML reader for the SQLite bus: live agents with heartbeat status, conversations grouped by id, per-message verification status color-coded, gap reports highlighted at the bottom. Reading the database is half a screen of code; the dashboard makes the next three problems debuggable.

3

Add a third agent and watch what happens

Two agents is a conversation. Three is a system. Add a third — call it Gamma, a writer that takes Alpha's research and Beta's data and produces a final artifact — and observe what breaks. That's where the topic ontology, the blackboard discipline, and the gap-report flywheel get pressure-tested. Nothing in the substrate has to change; the cracks that appear are the design questions worth answering next.

Two primitives, eight pros, four honest limits, three moves. That's the entire system. The interesting work from here is application logic — what topics matter, what claims matter, who escalates to whom — not infrastructure.
