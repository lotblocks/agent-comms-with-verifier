Design specification · No. 02

The verifier primitive.

In-band claim validation for Hyperagent skills. Adversarial review that prevents bad output from reaching the user, and turns its own failures into permanent skill improvements.

READ  15 min STATUS  Design — not yet implemented UPDATED  2026·05·18 RELATED  Ref. No. 01 (Agent comms)

When a Hyperagent skill executes today, the result is taken at face value. The output goes back to the caller. If the skill did the wrong thing, said the wrong thing, or skipped a step — there is no second pair of eyes between the failure and the user. The verifier primitive is that second pair of eyes.

This document specifies the new tool, the contract between builder and verifier, the configuration of the verifier subagent, the gap-report flywheel that turns unverifiable claims into skill improvements, and the failure modes the implementation must defend against. It is written to be the design doc that precedes a build — concrete enough that an engineer can implement it, abstract enough to survive minor revision.

The verifier is not a replacement for rubrics. Rubrics score outputs after they ship — they generate eval data and surface patterns over many runs. The verifier runs before output ships and can refuse to ship it. They are complementary: verifier verdicts can themselves feed rubric data, but their jobs are different. Rubrics learn what good looks like. The verifier enforces it.

§ 01 — MotivationThe review gap

Two constraints govern almost every agentic system: planning and reviewing. Hyperagent already has strong support for planning — the Working Doc, plan-mode, the discipline of writing checkbox tasks before execution. What it does not have is in-band review. When a skill returns a result, the orchestrator either trusts it or doesn't, but there is no separate process whose only job is to interrogate the result and either accept it or send it back.

The cost of this gap shows up in three places. First, silent wrong answers — a skill returns "success" but did something subtly off; the user finds out later, or never. Second, undocumented assumptions — skills accumulate implicit knowledge in their authors' heads that never makes it into the documentation, so the skill works for them and fails for everyone else. Third, accumulated drift — small misalignments between skill behavior and skill documentation compound over months until the docs are decorative.

The verifier closes all three. It catches silent wrong answers by re-deriving the result from independent evidence. It surfaces undocumented assumptions when it cannot verify something the skill apparently did. And it provides a continuous correction signal that keeps documentation honest.

A skill that has been verified one hundred times is not the same skill as one that has been executed one hundred times. The first has been pressure-tested; the second has been hoped about.

§ 02 — The primitive at a glanceBuilder, verifier, gap report

The verifier primitive is exposed as a single new tool, RunSkillVerified. It wraps the existing RunWithCredentials path. From the caller's perspective the surface area is small: pass the same arguments you would pass to RunWithCredentials, plus the user's plain-language intent and an optional verifier configuration. Get back the skill's result plus a verification record.

Behind the surface, a deliberate sequence unfolds. The skill runs. Its output, the user's intent, and the skill's own documentation are handed to a verifier subagent — a different model, different tool budget, different identity. The verifier decomposes what should be true into a list of atomic claims and validates each one with its own evidence-gathering. Three outcomes are possible per claim: pass, fail, unverifiable. The aggregate produces one of three verdicts at the run level: verified, failed, or partial.

On failure, the verifier composes a remediation prompt and the builder is re-invoked. On unverifiable claims, the verifier emits a gap report — a structured suggestion of what the skill's documentation should say so that next time, the same claim becomes verifiable. The gap report flows back into the existing UpdateSkillAndScripts draft flow, where the user reviews and accepts.

The full loop: caller invokes RunSkillVerified → builder produces output → verifier decomposes and validates → on fail, remediate; on unverifiable, draft skill-doc improvements via the existing UpdateSkillAndScripts flow. The flywheel is the dashed path back to documentation.

§ 03 — Tool APIRunSkillVerified

The new tool sits alongside RunWithCredentials in the skill execution family. Its signature is deliberately a small extension of what callers already know.

// Tool: RunSkillVerified
// Wraps RunWithCredentials with adversarial verification.

interface RunSkillVerifiedParams {
  skillName: string;          // same as RunWithCredentials
  command: string;            // same as RunWithCredentials
  intent: string;             // REQUIRED: user's goal in plain language
  verifier?: {
    model?: string;            // default: pick a different family than builder
    strictness?: "low" | "medium" | "high";   // default: "medium"
    maxAttempts?: number;     // default: 2 (one build + one re-prompt)
    claimGuidelines?: string; // skill-specific decomposition hints
    budgetUsd?: number;       // default: 0.50 per attempt
    timeoutSec?: number;      // default: 120
  };
}

interface RunSkillVerifiedResult {
  result: any;                 // the skill's final output
  attempts: number;             // how many builder invocations
  verification: {
    status: "verified" | "failed" | "partial";
    claims: Claim[];          // every claim with its verdict
    gapReport?: GapReport;     // present iff unverifiable claims exist
    verifierModel: string;
    durationMs: number;
    costUsd: number;
  };
}

Parameter detail

intent required · string
The user's plain-language goal. Not the prompt to the skill — the underlying goal. The verifier uses this to know what "correct" actually means. Without it, the verifier can only check internal consistency, not whether the skill did the right thing. Example: "create a payment intent for the new annual plan that succeeds in test mode".
verifier.model optional · string default: cross-family
Which model runs the verifier. The default rule is different family from the builder — if the builder is on Opus, the verifier defaults to GPT or Gemini. The point is independent failure modes. If the builder and verifier are the same model, the verifier becomes a sycophant.
verifier.strictness "low" / "medium" / "high" default: medium
How many claims the verifier should produce per output. low ≈ 3-5 claims, structural only. medium ≈ 5-10 claims, structural + behavioral. high ≈ 10+ claims, all five types. Strictness can be set per skill in skill metadata so the caller doesn't have to specify every time.
verifier.maxAttempts number default: 2
Maximum total builder invocations including the first. Default 2: one build, one remediation attempt if the first fails. Setting to 1 disables remediation; the verifier only reports. Setting above 3 is generally a sign that the skill is broken and re-prompting won't save it — better to surface the failure.
verifier.claimGuidelines optional · string
Free-form text appended to the verifier's system prompt with skill-specific decomposition hints. For a Stripe payment skill, this might be "always check that the customer ID in the response matches the one in the request". Often lives in skill metadata once stabilized.
verifier.budgetUsd number default: 0.50
Hard cap on verifier cost per builder attempt. The verifier may make tool calls (read files, hit read-only APIs) — this limits the blast radius if the verifier's evidence gathering goes long. On overrun, verification ends; remaining claims are marked unverifiable with reason budget exceeded.
verifier.timeoutSec number default: 120
Wall-clock timeout per verification pass. Same failure semantics as budget overrun.

§ 04 — The claim contractWhat gets verified, and how

A claim is a single, independently testable assertion about the skill's output, its side effects, or the world the skill touched. Claims are the unit of verification. The whole architecture turns on getting them right: too coarse and you cannot pinpoint what failed; too fine and the verifier spends its budget on trivia.

Every claim has the same shape, regardless of type:

interface Claim {
  id: string;                 // e.g. "claim_001"
  type: ClaimType;              // see below
  statement: string;          // human-readable assertion
  evidenceRequired: string;   // what would prove this true
  evidenceCollected?: any;    // what the verifier actually saw
  verdict: "pass" | "fail" | "unverifiable";
  confidence: number;          // 0–1
  reasoning: string;          // why the verifier chose this verdict
}

There are five claim types. Most outputs have a mix of two or three. The five exist because different verdicts require different evidence-gathering strategies, and the verifier prompt needs to know which strategy to apply.

Existential

Existential

Asserts that a thing exists. Resource, file, record, artifact. Cheapest to verify — single read.

statement: "the file at /workspace/report.pdf exists"
evidence: stat(path) returns ok
Structural

Structural

Asserts that a thing has a specific shape. Schema match, field presence, type correctness. Verified by parsing.

statement: "the JSON has field 'customer_id' of type string"
evidence: parsed[customer_id] is string
Behavioral

Behavioral

Asserts that a thing does something. Function returns X for input Y. Most expensive — requires execution.

statement: "GET /api/users/42 returns 200"
evidence: actual HTTP call, status 200
Factual

Factual

Asserts that something is true about the world. Citation, date, name, identifier. Verified against an authoritative source.

statement: "the company HQ is in Tallinn"
evidence: official website / corp registry
Semantic

Semantic

Asserts that the output means what was asked. Intent alignment. Hardest to make objective — verifier judges, with reasoning.

statement: "the report addresses the user's question about Q2 churn"
evidence: reading the report, mapping to intent
Negative

Negative (optional)

Asserts that something did not happen. PII leaked. Wrong endpoint called. Sensitive field exposed. Verified by absence.

statement: "no PII fields appear in the response body"
evidence: scan for SSN, email, phone — none found

How claims are produced

The verifier subagent receives three inputs and produces the claim list itself: the skill's documentation (what is this skill supposed to do?), the user's intent (what did the user actually want?), and the builder's output (what happened?). It is explicitly instructed to produce claims as the intersection of "what the skill promised" and "what the user asked for". Claims that fall outside both are noise.

Strictness governs how many claims to produce. A medium-strictness verification of a Stripe payment skill might yield: one existential (PaymentIntent exists), three structural (id format, amount type, currency code), two behavioral (test mode flag respected, idempotency key honored), and one negative (no live-mode keys in the response). Seven claims, seven verdicts, one aggregate verdict.

The aggregate rule is conservative. verified requires all claims pass. failed requires at least one claim fail. partial means no failures but at least one unverifiable. Partial verifications return the skill's output but flag the gap and produce a gap report.

§ 05 — Verifier configurationHow the subagent is set up

The verifier runs as an isolated subagent spawned through the existing Agent tool. It is not a free-form conversation. It receives a structured prompt, a restricted toolset, a budget, and a contract for its output format. Everything about the verifier is set up to produce independent evidence, not consensus.

Identity & model

The subagent description follows the existing convention: Name | Role | Task. For verifications, a recommended name is something neutral and observant — Sentinel, Veritas, Lens. Example: Sentinel | Claim Verifier | Validate the output of stripe.create_payment_intent against the user's intent.

Model selection follows the cross-family default unless overridden. A useful pairing matrix:

// Recommended verifier pairings
builder: opus      → verifier: gpt-5.5  | gemini-pro
builder: gpt-5.5   → verifier: opus     | claude-sonnet
builder: sonnet    → verifier: opus     | gpt-5.5
builder: haiku     → verifier: sonnet   // for cheap, opt-in

Tool budget

The verifier has a read-only toolset. It can:

  • Read files (Read, Glob, Grep)
  • Search the web for factual claims (WebSearch, ExaSearch)
  • Call read-only integration actions (GETs on connected services)
  • Inspect the skill's documentation (GetKnowledgeDetails)

It cannot:

  • Write or edit files
  • Execute scripts (RunWithCredentials, Bash with side effects)
  • Call mutating integration actions (POST / PUT / DELETE)
  • Spawn further subagents
  • Use RunSkillVerified — no recursive verification

Restricting the verifier to reads is what makes the architecture safe. The verifier can investigate but cannot change the world, which means a misbehaving verifier degrades to noise rather than damage.

System prompt skeleton

You are the verifier. You did not run the skill.
You have NOT seen the user's chat. You have received:

  - the skill's documentation
  - the user's plain-language intent
  - the skill's output

Your job is to decompose what should be true into a list of
atomic claims and produce a verdict for each. You are not
trying to be helpful. You are trying to be RIGHT.

You produce claims as the intersection of:
  (a) what the skill's documentation says it does, AND
  (b) what the user's intent says they wanted.

Claims outside both are noise. Do not produce them.

For each claim:
  - state it precisely
  - state what evidence would prove it
  - gather that evidence using your read-only tools
  - render a verdict: pass, fail, or unverifiable
  - confidence and reasoning are mandatory

If a claim is unverifiable, that is a finding, not a failure.
Report it with: WHY you cannot verify it, and WHAT the skill
documentation would need to say for it to become verifiable
next time. This becomes the gap report.

Strictness target: {LOW|MED|HIGH} — produce N±2 claims.

§ 06 — The remediation loopWhat happens on failure

If the verifier returns failed and attempts < maxAttempts, the builder is re-invoked with a structured remediation prompt. This is not "try again". It is targeted, claim-by-claim feedback.

// Remediation prompt template

Your previous output was reviewed by an independent verifier.
The following claims FAILED:

{for each failed claim:}
  - {claim.statement}
    Why it failed: {claim.reasoning}
    Evidence the verifier saw: {claim.evidenceCollected}
    What needs to change: {derived from claim type}

The following claims could NOT be verified:

{for each unverifiable claim:}
  - {claim.statement}
    Why not: {claim.reasoning}
    {if relevant: what additional information you should include}

Run the skill again. Do not change the user's original request.
Address each failed claim. For unverifiable claims, include
the evidence the verifier would need to validate them.

Original command: {original.command}
Original intent: {original.intent}

The builder runs again. The output is verified again. If failed persists past maxAttempts, the tool returns the latest output with status: failed, the verification record, and an exception-style flag. The caller decides whether to surface this to the user, escalate to a human channel, or fall back to a different strategy.

The remediation loop is deliberately bounded. Two attempts is the default because a skill that fails twice on the same claim is almost certainly not going to pass on the third. The right response is to surface the failure, not to keep spending. The exception is calibrated experimentation — for skills under active development, a higher maxAttempts is fine; the failure pattern itself is data.

§ 07 — The gap-report flywheelHow unverifiable claims become skill improvements

The most distinctive part of the verifier primitive is what happens to claims marked unverifiable. These are not failures — they are documentation gaps. Some examples of legitimate unverifiables:

  • "The skill is supposed to deduct sales tax, but I cannot confirm from the output whether tax was applied or not — the skill's response shape doesn't surface it."
  • "The skill should be idempotent on the user's request key, but the documentation does not specify which field carries the key — I cannot test idempotency."
  • "The skill returned an artifact ID but I have no way to confirm that ID corresponds to the file the user asked for — there's no naming convention documented."

Each of these is a real, latent problem in the skill itself. The verifier names it. The gap report packages it.

interface GapReport {
  skillId: string;
  unverifiableClaims: Claim[];
  proposedImprovements: Improvement[];
  summary: string;        // 1-2 sentences for the user
}

interface Improvement {
  claimId: string;        // which claim this addresses
  target: "documentation" | "output_schema" | "script";
  currentText?: string;    // what's there now (if any)
  proposedText: string;    // what should be added/changed
  rationale: string;       // why this would help
  confidence: number;       // 0–1, gates auto-surfacing
}

The gap report flows into Hyperagent's existing draft-card flow. It does not auto-apply. The implementation auto-generates a UpdateSkillAndScripts call with saveMode: "draft", populated with the proposed documentation additions or schema changes. The user sees the inline draft card — same flow they already know — reviews the suggestion, edits if needed, and accepts or rejects.

This matters because the gap-report flywheel is only as good as the discipline that keeps low-quality suggestions out of the skill. The draft card is the gate. Confidence-gated emission (e.g. confidence >= 0.7) keeps the volume manageable. A skill that's seen heavy use accumulates a steady, curated stream of improvements; a skill that's barely used produces few suggestions because it's barely producing claims to be unverifiable about.

A skill improves itself by being asked questions it can't answer. The verifier asks the questions; the gap report names them; the user decides whether to make them answerable next time.

§ 08 — Failure modesWhat breaks, and how we defend

The verifier primitive introduces its own failure modes on top of the ones inherent to the skill it wraps. The implementation must defend against all six.

01

Infinite remediation loops

Builder fails. Verifier remediates. Builder fails the same way. Verifier remediates again. The loop never converges; the budget burns.

MitigationHard cap via maxAttempts (default 2). Per-attempt budget cap via budgetUsd. On exhaustion, return latest output with status: failed and the verification record — the caller decides what to do. Same-claim-failure detection: if the same claim ID fails twice with the same reasoning, mark the skill as structurally broken for this intent and skip further attempts.
02

Verifier hallucination

The verifier rejects valid output. It claims the JSON is missing a field that is actually there, or that an API returned 500 when it returned 200. Worse, it does this confidently. The user sees a false rejection.

MitigationMandatory evidenceCollected on every fail verdict — the verifier must show its work. Logged evidence is auditable; the user can see exactly what the verifier saw and judge whether the verdict was fair. Periodic offline rubric-based evals on a sample of verifications spot patterns of false rejection. Confidence threshold: failures with confidence below 0.6 are downgraded to unverifiable with a flag.
03

Claim decomposition collapse

The verifier produces one vague claim — "the output looks correct" — instead of the seven specific claims it should have. The verdict is structurally meaningless. Worst case, it always passes; the verifier becomes ceremonial.

MitigationMinimum claim count enforced by strictness level: low ≥ 3, medium ≥ 5, high ≥ 10. Verifier output is validated against schema before being accepted; a sub-minimum count triggers a single retry with stricter prompting; a second sub-minimum count marks the verification as degenerate and surfaces a warning. The skill's claimGuidelines field is the place to fix this once observed.
04

Same-family bias

Builder and verifier are both Opus. They share training data, share failure modes, agree on wrong answers. The verifier is technically independent but functionally identical. False confidence everywhere.

MitigationCross-family default is enforced unless explicitly overridden, with a warning logged when overridden. Where cross-family is not possible (cost, availability), at minimum cross-prompt: the verifier's system prompt is structurally different from the builder's, with explicit instruction to disagree with the builder when evidence supports it.
05

Gap-report noise

Every verification produces a slew of low-quality skill-doc suggestions. The user is buried under draft cards. They start ignoring them. The flywheel that was supposed to improve the skill becomes a source of friction.

MitigationConfidence gate on Improvement emission — default 0.7, configurable per skill. Same-suggestion deduplication: if the same proposed improvement appears across three verifications for the same skill, it is upgraded in priority and surfaced as a single card with the count, not three. The user can silence a class of suggestion for a skill, and the silence is honored.
06

Verifier latency dominates

The skill takes 4 seconds. The verifier takes 30. Users feel the wrap is the cost, and start setting maxAttempts: 1 or disabling verification — losing the value that the primitive was supposed to deliver.

MitigationParallelize claim verification — claims are independent by construction, so the verifier evaluates them concurrently rather than serially. Aggressive caching of evidence collection (the same GET /api/users/42 doesn't re-run within a verification pass). Surface verification cost and time in the result so callers can tune. Async mode for very long verifications: RunSkillVerified returns the builder's output immediately and verifies in the background, posting a follow-up card if the verification flags something.

§ 09 — What's deferredOpen questions for v1

Some interesting questions don't need answers before the first build. Listing them so they don't get rediscovered as bugs.

Multi-skill verification

If a high-level task chains three skills together, does the verifier wrap each call or the whole chain? V1 answer: each call. V2 might support chain-level verification with cross-skill claims.

Human-in-the-loop verification

For high-stakes skills (anything that moves money, deletes records, posts publicly), should the verifier route to a human for one or more claims? V1: no — verifier is fully automated. The pattern exists as a future extension via a special claim type human_attestation.

Verifier of verifiers

Who watches the watchers? V1 says rubric-based offline eval is sufficient — sample verifications and score the verifier's verdicts. No live nested verification. The recursive case is explicitly disallowed (no RunSkillVerified from inside a verifier).

Per-skill verifier defaults

Eventually the skill metadata should carry default verifier configuration: model preference, strictness, claimGuidelines, cost cap. V1 supports skill-metadata defaults but doesn't require them — every skill can be verified with the global defaults out of the box.

Streaming verification

For long-running skills that produce intermediate output, the verifier could verify incrementally rather than wait for completion. V1: no streaming. Verification is end-of-skill only. Streaming is a natural follow-on once the synchronous version is stable.

Integration with rubrics

Verifier verdicts are themselves a signal — they should feed Hyperagent's rubric/eval system as training data for skill quality scoring. V1 logs verifications in a structured form; the rubric integration is a separate workstream.

§ 10 — SummaryWhat v1 ships

The minimum first version is bounded. Six things, no more:

  1. The RunSkillVerified tool with the API specified in §03.
  2. The five-claim-type contract from §04, with negative as optional.
  3. A verifier subagent spawned via the Agent tool, with the read-only toolset from §05 and cross-family model default.
  4. The bounded remediation loop from §06, maxAttempts default 2.
  5. The gap report → UpdateSkillAndScripts draft-card flow from §07, with confidence gating at 0.7.
  6. Defenses for all six failure modes from §08.

Everything in §09 is deferred. The aim of v1 is to prove the primitive works, not to make it universal. Two well-chosen pilot skills — one synchronous and predictable (e.g. a Stripe operation), one less so (e.g. a research task) — would surface most of the edge cases before scaling to the rest of the catalog.

The verifier doesn't make skills smarter. It makes the gap between what a skill claims and what it does visible — and gives that visibility a place to land. Over time the skills get better not because they were rewritten, but because their documentation finally tells the truth.
