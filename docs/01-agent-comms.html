An architecture primer · No. 01

Agents talking to agents.

A conceptual architecture for true bidirectional agent communication — beyond the orchestrator-worker pattern, toward systems where peers coordinate without a conductor.

READ  14 min UPDATED  2026·05·18 STATUS  Reference design

The default agent system is hierarchical. A planner agent decomposes work, dispatches subagents, collects their reports, synthesizes. It works well, scales reasonably, and is easy to reason about. It is also profoundly limiting.

Hierarchical systems treat every coordination question as a return value. The orchestrator asks; workers answer; the orchestrator decides. Workers never address each other. They cannot negotiate, escalate sideways, hand off, or form ad-hoc coalitions. They cannot say "actually, the agent next to me has the context for this — let me hand it over". The orchestrator is a bottleneck for every joint decision, and joint decisions are most of the interesting work.

True agent-to-agent communication means three things at once: agents have stable identities other agents can address, the transport supports both broadcast and direct messaging, and the runtime has enough discipline to keep the whole thing from devolving into a babbling mob. Building it well is mostly about getting eight primitives right.

§ 01 — FoundationsThe eight primitives

Any system where independent agents exchange messages needs to settle these eight design questions before a single message flows. Skip any of them and you'll rebuild it later under pressure.

PRIMITIVE 01

Agent identity

Each agent has a stable, unique ID and a human-readable name. The ID survives restarts; the name is what humans (and other agents) reference. Without stable identity, you cannot route, log, or revoke.

PRIMITIVE 02

Addressing

How does Agent A reach Agent B? By ID (precise), by name (friendly), or by role (e.g. any:researcher)? Most mature systems support all three, with a registry that resolves names and roles to IDs.

PRIMITIVE 03

Message envelopes

Every message carries metadata: from, to, topic, correlation_id, timestamp, ttl, and a typed payload. The envelope is what makes loop prevention, tracing, and replay possible.

PRIMITIVE 04

Topics & channels

Logical groupings that let agents subscribe by subject rather than by sender. project.thrift.brand is a topic; agents interested in brand decisions subscribe; everyone else stays quiet. Topics are how broadcast scales.

PRIMITIVE 05

Delivery semantics

At-most-once (fast, lossy), at-least-once (durable, may duplicate), or exactly-once (expensive, complex). Pick deliberately. For agent coordination, at-least-once plus idempotent handlers is the right default.

PRIMITIVE 06

Persistence

Are messages durable across crashes, or ephemeral? Durable transports survive restarts but cost more and complicate semantics. Ephemeral is fine for chatter; anything triggering side effects should be durable.

PRIMITIVE 07

Loop prevention

Without it, two polite agents will reply to each other forever. The standard mitigations are TTLs on every envelope, max-hop counters, conversation-ID tracking, and per-agent rate limits. Bake them into the transport, not the agents.

PRIMITIVE 08

Observability

You will not be able to debug this from logs alone. You need a message timeline, a conversation tree view, and the ability to replay a single conversation. Build the dashboard before you build the third agent.

§ 02 — TopologiesFour shapes of coordination

Every multi-agent system is a variant of one of these four topologies, or a deliberate combination of two. The names are old — they come from distributed systems and AI research from the 1980s onward — but the tradeoffs are still the ones that matter.

Orchestrator – Worker

One agent decomposes work and dispatches it. Workers never address each other; everything routes through the top.

Best for: bounded tasks, low coordination

Peer-to-Peer Mesh

Every agent knows how to address every other agent directly. No central node. Connections grow as n² — fine for 5 agents, painful at 50.

Best for: small fixed teams

Shared Blackboard

Agents read from and write to a common state. No one addresses anyone — coordination happens through observed change. Simple, but every agent has to poll or subscribe.

Best for: convergent reasoning, planning

Pub-Sub Event Bus

A central transport routes messages by topic. Senders don't know receivers; receivers don't know senders. Scales beautifully, but the bus is a single dependency to operate.

Best for: mixed broadcast + direct, larger systems

For the mix-mode you described — some broadcast, some direct — pure orchestrator-worker is out (no agent-to-agent), pure mesh is out (doesn't broadcast cleanly), and pure blackboard is out (no addressed messages). The working answer is a pub-sub bus that also supports direct addressing: agents publish to topics for broadcast, and use to: agent_id envelopes for direct. The bus handles both with the same primitives.

§ 03 — Reference architecturesThree ways to build it

Below are three concrete designs that all support broadcast plus direct. They differ on what infrastructure they assume, how fast messages move, and how much you have to build. None of them is wrong; they fit different stages of seriousness.

Option A — Hyperagent-native (Slack + shared docs)

Each agent is a named Hyperagent agent. Slack channels serve as broadcast topics; agents subscribed to a channel receive every message. For direct messages, agents post into another agent's thread via the platform's Slack/DM tools. A shared project document holds durable state — the blackboard. Schedules trigger periodic check-ins so nothing is purely event-driven (which Slack doesn't support natively for agent-to-agent).

This is the lowest-cost path: zero infrastructure, fully human-visible, works today. The price is latency (seconds to a minute), low throughput, and Slack's rate limits as a hard ceiling. Good for 3–5 agents coordinating on slow-moving knowledge work.

Option B — Hybrid (custom message-bus skill)

Build one skill that wraps a real message bus — Postgres LISTEN/NOTIFY is the pragmatic choice if you already have Postgres; Redis pub-sub if you have Redis. The skill exposes three actions: publish(topic, payload), subscribe(topic) → messages, and direct_send(agent_id, payload). Every Hyperagent agent that needs to participate has this skill enabled.

Agents still appear as normal named agents to humans. Internally they talk through the bus. State that needs to be durable beyond a message lives in a shared document. Slack stays — but only as the human escalation channel, not as the inter-agent transport. Sub-second latency, durable, fits the platform cleanly, but requires building and operating the skill plus its backing store.

Option C — Full custom bus (NATS / Redis Streams)

Stand up a proper messaging system — NATS or Redis Streams — and run a registry service alongside it. Agents become long-running processes that hold open subscriptions; messages arrive in milliseconds. This is what production agent systems look like at scale.

The cost is real engineering: you're now operating a message broker, a registry, an observability stack, and the agent runtime as four separate concerns. Worth it for high-frequency systems (market-making agents, real-time monitoring fleets); overkill for almost everything else.

§ 04 — Failure modesFive ways this breaks

These are not edge cases. Every multi-agent system that runs in production hits each of them; mature systems are simply the ones that designed for them up front rather than after the second outage.

01

Infinite reply loops

Agent A asks Agent B for clarification. B doesn't have it either and asks A. A still doesn't have it. Repeat forever, in milliseconds, until something runs out of money or memory. The polite-agents-in-a-loop pattern is the most common way these systems fail.

MitigationEvery envelope carries a conversation_id, a hop_count, and a ttl. Drop messages over a hop limit (e.g. 8). Track per-conversation message counts and circuit-break at a threshold. Make these properties of the transport, not opt-in for each agent.
02

Broadcast storms

One agent publishes "state changed" to a topic. Twelve subscribers each respond by publishing their own derived state. Those derivations trigger more subscribers. A single event becomes hundreds of messages in seconds.

MitigationPer-topic rate limits on the bus. Dedupe by content hash within a short window. Require subscribers to debounce their own responses. Prefer pull-based reads (agents query state when they need it) over push-on-every-change.
03

Stale state

The blackboard says the brand color is teal. Agent A reads it, starts a long task. Meanwhile Agent B updates the blackboard to coral. A finishes, writes its result based on the now-wrong assumption. Nobody noticed.

MitigationVersion every shared state object. Reads return the version; writes are conditional on version (optimistic concurrency). Assign a single-writer-per-key where possible — only the Brand agent writes brand state. Other agents propose changes via messages, not direct writes.
04

Ghost agents

An agent process dies but its subscriptions remain in the registry. Other agents keep sending it work that goes nowhere. The system slowly fills with addressed envelopes nobody is reading. Detection takes hours; trust takes longer to rebuild.

MitigationHeartbeats. Every active agent pings the registry on an interval (e.g. 30s). Subscriptions for agents that miss N heartbeats are auto-revoked. Messages sent to dead agents are dead-lettered, not silently dropped. A status dashboard shows live/stale/dead per agent.
05

Silent deadlocks

Agent A is blocked waiting for Agent B's response. B is blocked waiting on Agent C. C is waiting on A. The system stops making progress but no error fires — every agent thinks it's just waiting patiently.

MitigationNo blocking waits. Every request-response pattern carries a timeout. On timeout, agents either retry with backoff or escalate to a human channel. Periodic "progress beacons" let the observability layer detect when a conversation has gone quiet for too long.

§ 05 — RecommendationThe architecture I'd build

If you're starting now, build Option B — the hybrid. It fits Hyperagent's existing model (named agents stay named agents, humans still see them in Slack and the UI), it gives you sub-second coordination where the native path would take a minute, and the build is bounded: one skill, one backing store, one shared document for durable state.

Four agents, one bus, one shared blackboard, Slack only at the human edge. Each agent subscribes to the topics it cares about and addresses peers directly when it has something specific to say.

The message envelope looks like this — kept deliberately simple, because every field has to earn its place:

// Message envelope · the contract between all agents
{
  "id": "msg_01HXYZ...",           // unique per message
  "conversation_id": "cnv_...",    // groups related msgs
  "from": "agent_001",
  "to": "agent_003",              // or null for broadcast
  "topic": "project.brand",        // or null for direct only
  "type": "request",               // request | response | event
  "reply_to": "msg_01HXYZ...",     // for responses
  "hop_count": 0,                   // incremented per relay
  "ttl_seconds": 300,               // drop after this
  "timestamp": "2026-05-18T19:00:00Z",
  "payload": { /* typed body */ }
}
The discipline isn't in the message format. It's in the rule that every agent honors hop_count, ttl_seconds, and conversation_id without exception — because the second one stops is the moment the whole system becomes unsafe.

§ 06 — PrototypeA minimum viable build

If you want to see this working in a week, here's the shortest credible path. Each step is small enough to finish in a sitting; together they get two agents truly talking — not via an orchestrator, not via a human in the middle.

1

Build the message-bus skill

Three scripts: publish.py, subscribe.py, direct_send.py. Use Postgres with a messages table and LISTEN/NOTIFY, or Redis pub-sub if you already operate Redis. The skill enforces hop limits and TTL at the transport layer so individual agents don't have to.

2

Stand up an agent registry

A small table mapping agent_id → name, role, subscriptions, last_heartbeat. Agents register on startup and heartbeat every 30 seconds. Anything that misses 3 heartbeats gets marked stale; sends to stale agents go to a dead-letter queue.

3

Create two named Hyperagent agents

Start with a deliberately asymmetric pair — say Researcher and Writer. Different system prompts, different responsibilities, both with the message-bus skill enabled. Each subscribes to one shared topic plus its own inbox.

4

Define two conversation patterns

One broadcast pattern — Researcher publishes a finding to project.research, Writer reads it asynchronously. One direct pattern — Writer DMs Researcher asking for a specific clarification, gets a typed response. Both go through the same bus with the same envelope.

5

Build the smallest possible dashboard

A single page that shows live messages grouped by conversation_id. You'll find every bug through this view. Don't add the third agent until you can read the first two's conversations the way you'd read a transcript.

6

Promote one shared document to blackboard

Pick one piece of state that both agents need to agree on — a project summary, a brief, a decision log. Make it versioned. Only one agent writes; others propose changes through messages. This is the discipline that keeps the system from drifting.

What you get

After step 6 you have a working substrate for true agent-to-agent communication. Two agents talk to each other without an orchestrator. Messages are durable, traceable, and loop-safe. Adding a third agent is a matter of giving it the skill and registering it — the bus, the envelopes, the failure-mode discipline are already there. From here, scaling to five or fifteen agents is a deployment exercise, not a redesign.

What you do not get for free: judgment about who should talk to whom about what. That's still a system-design question every project answers differently. The architecture above is the substrate. The interesting work is the topic ontology — what subjects matter, who cares about each, how decisions get escalated. Get that right and the system feels alive. Get it wrong and you have a chat room where nothing decides anything.

Agent-to-agent communication isn't a tooling problem. It's a discipline problem. The tooling is straightforward once you've made the hard decisions about identity, addressing, and what counts as a conversation worth having.
