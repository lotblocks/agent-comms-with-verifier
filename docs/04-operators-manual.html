§ 01 — OverviewThe system at a glance

Three primitives. One substrate. Two thousand lines of code. A working model for multi-agent systems where every claim is checked, every conversation is auditable, and the system gets smarter every time it runs.

The system answers a question multi-agent frameworks usually leave open: how do agents trust each other? Hierarchical orchestrators paper over the question by serializing every decision through a single planner. Real peer-to-peer systems can't afford that — the planner becomes the bottleneck, the bug, and the single point of failure.

This system solves it by inverting the trust contract. Workers don't ask to be trusted; they prove it. Every reply carries a structured verification record: which claims about the output were checked, by what model, with what evidence, at what confidence, with what gaps still open. Receivers inspect that record and decide whether to accept. Over many runs, gap reports become permanent skill documentation improvements, and per-agent reputation scores let the system route around flaky workers — a flywheel where every interaction is signal, not noise.

An agent system isn't useful because it's clever; it's useful because the next agent in the chain can trust the result. This system ships the trust as data.

§ 02 — ComponentsThe five components

Component 01

The Bus

SQLite-backed peer-to-peer message broker. Envelopes carry from/to, topic, conversation lineage, hop count, TTL, payload. Atomic claim-locking on receive so worker pools don't double-process. Loop prevention baked in.

demos/comms-with-verifier/bus.py
Component 02

The Verifier

Wraps any builder with claim validation. Decomposes output into atomic claims, verdicts each, remediates on failure, emits gap reports for unverifiable claims. Three pluggable backends (mock, Anthropic, Replicate) behind the same verify() signature.

skills/verifier/run_skill_verified.py
Component 03

The Memory

SQLite-backed durable learning store. Three scopes (global, agent.<id>, skill.<name>). Gap reports stored as memories; reinforced on repeat. Reputation tracked per-agent. Rendered as system-prompt hints for the LLM verifier.

demos/comms-with-verifier/agent_memory.py
Component 04

The Agents

Three reference agents — Alpha (requester), Beta (worker), Gamma (intermediate worker that talks to other workers peer-to-peer). All build on shared lifecycle helpers; verify-before-publish discipline is universal.

demos/comms-with-verifier/agent_*.py
Component 05

The Dashboard

Static HTML observability surface. Reads the bus DB, renders live agents, conversations grouped by chain, per-message verification banners, merged gap reports. Generated by a single Python script.

demos/comms-with-verifier/build_dashboard.py
Supporting

Two more files worth knowing

verification_chain.py — walks nested verification records and emits a chain summary (weakest-link trust, total cost, merged gap reports). agent_lifecycle.py — graceful shutdown plumbing for production agents.

verification_chain.py · agent_lifecycle.py

§ 03 — ArchitectureHow the pieces fit

The diagram below shows the whole substrate. The bus is the only shared dependency. Memory is independent — wiped, replaced, or persisted on its own schedule. Agents own their own process state. The verifier lives inside each worker, private but composable.

Three agents on a shared bus. Each worker has its own verifier (any backend), reads and writes to a shared memory store, and replies with a verification record. The dashboard renders both bus and memory; Slack sits at the human edge for escalation only.

§ 04 — FlowOne full request, end to end

The compact version of what happens when Alpha asks for verified work, told as nine steps. The full sequence diagram lives in Reference No. 03; this is the abridged version with concrete file references.

  1. Alpha sends a request. Either direct (send_direct), broadcast (publish_to_role), or reputation-aware pick. A new conversation_id is minted; envelope carries TTL and hop_count.
  2. Beta receives. Polls bus.receive(subscriptions=[...]); atomic claim-locking ensures one replica gets the message.
  3. Beta consults memory. If AGENT_MEMORY_DB is set, reads per-skill memories and appends them as MEMORY HINTS to skill_doc.
  4. Beta calls the verifier orchestrator. Subprocess to run_skill_verified.py with the chosen backend. The orchestrator runs the builder, decomposes into claims, remediates if needed.
  5. Verifier returns a record. Status (verified/failed/partial), all claims with verdicts, optional gap report, model and cost metadata.
  6. Beta stores memory. Gap reports become per-skill memories (deduped, reinforced). Reputation memory is bumped (success or failure).
  7. Beta builds chain_summary. Single-hop case for Beta; intermediate workers like Gamma nest their upstream verification first, then build chain_summary across all hops (weakest-link status, total cost, merged gap reports).
  8. Beta replies. send_direct to Alpha's id with response payload containing verification + chain_summary.
  9. Alpha trust-checks. Reads chain_summary.chain_status (end-to-end weakest link) — not local verification.status — and decides whether to accept.
The trust handoff is the entire point. Alpha doesn't have to take Beta's word for it. Alpha sees the claims, evidence, model, cost, and gap reports. Trust travels as data.

§ 05 — MemoryHow memory works (and persists)

Memory is the flywheel. Every verification produces signal — gap reports name documentation gaps, failed claims name brittleness patterns, successful runs accumulate reputation. The memory store turns that signal into durable knowledge that biases future runs.

Three scopes

ScopePurposeExample content
global Shared facts across the whole system "Skills returning PII must include a redaction summary"
skill.<name> Per-skill knowledge — gaps, failure patterns "demo-compute-total often omits timestamp on first attempt"
agent.<id> Per-agent reputation, preferences "agent_beta_flaky: 0/3 success rate"

The flywheel — closed loop

Step 1: First run on a skill
  Verifier emits gap_report (unverifiable claim)
       ↓
  Worker calls memory.store_gap(skill_name, gap)
       ↓
  Step 2: Second run on the same skill
  Worker calls memory.claim_guidelines_for_skill(skill_name)
       ↓
  Returns rendered hints appended to skill_doc:
       "KNOWN DOCUMENTATION GAPS FOR THIS SKILL: ..."
       ↓
  LLM verifier sees hints in its system prompt
       ↓
  Produces more precise claims that target the known gap
       ↓
  Either the gap is now verifiable (signal: docs fixed)
  OR it's still unverifiable (signal: reinforce memory)
       ↓
  Step 3: store_gap deduplicates by (scope, subject, statement)
  Same observation → bump use_count + importance by 0.05 (capped at 1.0)
  Not a duplicate row — a reinforced one
       ↓
  Result: heavy-use skills accumulate a curated knowledge base of their own quirks

Persistence

Memory lives in its own SQLite file, deliberately separate from the bus DB. The bus represents one conversation epoch; memory represents the system's accumulated knowledge across all conversations. Wipe the bus to start fresh — memories survive.

PropertyBus DBMemory DB
File_demo_bus.sqlite_agent_memory.sqlite
LifecyclePer-run; wiped between demosLong-lived; survives across runs
OwnerThe bus libraryMemoryStore class in agent_memory.py
Enabled viaBus constructor (always)AGENT_MEMORY_DB env var or --memory-db flag

Key MemoryStore methods

store(scope, category, content, subject, importance)        // generic
store_gap(skill_name, gap_report)                            // dedup + reinforce
store_failure_pattern(skill_name, claim_id, ...)             // for "X failed because Y, fixed by Z"
store_reputation(agent_id, success)                          // 1 bump per run
reputation_summary(agent_id) → {successes, total, success_rate}
claim_guidelines_for_skill(skill_name) → string (for system prompt)
recall(scope, category?, subject?, limit) → list[Memory]

§ 06 — DispatchThree routing modes

The same agent can be addressed three ways. Each has different semantics and best-fit scenarios.

1. Direct addressing

--target-id agent_beta_1

  • Caller picks a specific replica by id
  • Message goes to that replica's personal inbox
  • Use when: pinning to a known agent for traceability or pinning experiments

2. Topic fanout

--target-role worker

  • Caller publishes to role.<role> topic
  • Any subscribed replica can claim (first-come)
  • Use when: load-balancing across worker pools — the bus's atomic claim-locking is the load balancer

3. Reputation-aware pick

--target-role worker --reputation-min 0.6

  • Caller queries memory; filters replicas by success_rate
  • Picks one trusted replica; falls back to topic-fanout if none
  • Use when: routing around known-flaky workers; high-stakes operations

Probation rule

min_reputation_samples: 3 (default)

  • Replicas with fewer than N samples get a probation pass
  • New agents aren't filtered out just because they haven't earned reputation yet
  • Configurable per call

Decision tree

Need to address a worker?

  Do you know the specific agent_id?
    ├── YES → use --target-id (direct addressing)
    └── NO
        Do you have a worker pool with replicas?
          ├── YES
          │   Do some replicas have a track record (reputation)?
          │     ├── YES, and quality matters → --target-role + --reputation-min
          │     └── NO, or all equivalent     → --target-role (topic fanout)
          └── NO, one worker → --target-id with its known id

§ 07 — BackendsWhich verifier to use

Backend Cost Quality Latency Setup When
mock free deterministic stub <1 ms none Offline tests; orchestration debugging; CI
llm (Anthropic) $0.003-0.015/call high — Claude Sonnet/Opus 2-10s ANTHROPIC_API_KEY Highest-stakes production; nuanced claim reasoning
replicate ~$0.0002/call good — Llama-3, Mistral, etc. 1-8s REPLICATE_API_TOKEN High-volume verification; cost-sensitive deployments

End-to-end tested

The Replicate backend has been verified end-to-end against the real API. A live call to meta/meta-llama-3-8b-instruct with the verifier system prompt returned a 7-claim JSON record that parsed correctly through replicate_verifier.py's logic into a VerificationRecord with status=verified. Cost for that call: ~$0.0002.

Adding a fourth backend

Any provider with a chat-completion API (OpenRouter, OpenAI, Groq, Together, etc.) can be added in ~150 lines following the pattern of replicate_verifier.py. Implement the same verify() signature, wire it into _load_verifier() in run_skill_verified.py, and declare the API key in the skill credential schema.

§ 08 — ModelsRecommendations for builder + verifier pairings

The verifier should be from a different model family than the builder. Same-family pairings risk shared blind spots — both models miss the same kinds of bugs. Different families have different failure modes, which is the whole point.

Recommended pairings

Builder Recommended verifier Backend Why
claude-opus gpt-4.5 or llama-3-70b llm or replicate Cross-family; verifier doesn't share Claude's reasoning quirks
claude-sonnet llama-3-70b-instruct replicate Strong, cheap independent reasoning
gpt-4.5 claude-sonnet or claude-opus llm Anthropic's reasoning often catches what GPT misses
llama-3-8b llama-3-70b or claude-haiku replicate or llm Step up to a stronger reasoner for verification
llama-3-70b claude-sonnet llm Cross-family check on big-Llama work
any (testing) mock mock Orchestration testing only — never production

Strictness and claim counts

The --strictness flag governs how many claims the verifier produces per pass:

  • low — 3-5 claims; structural only. Fast, cheap, for low-stakes outputs.
  • medium (default) — 5-7 claims; structural + behavioral. The right default.
  • high — 10-12 claims; all five types. For high-stakes operations (payments, deletions, public posts).

§ 09 — UsageHow to use the system

Installation

git clone https://github.com/lotblocks/agent-comms-with-verifier.git
cd agent-comms-with-verifier
# No dependencies beyond Python 3.9+ stdlib

The five demos, in order of complexity

# 1. Two-agent — Alpha asks Beta directly
cd demos/comms-with-verifier
python3 run_demo.py

# 2. Three-agent — Alpha → Gamma → Beta peer-to-peer
python3 run_three_agent_demo.py

# 3. Concurrent worker pool — 5 alphas × 2 gammas × 2 betas
python3 run_concurrent_demo.py

# 4. Learning — same task twice; memory accumulates
python3 run_learning_demo.py

# 5. Reputation-weighted dispatch — flaky replica gets routed around
python3 run_reputation_demo.py

Three common production patterns

Pattern A — Single worker

# Beta serves forever; Alpha clients arrive over time
python3 agent_beta.py --db ./bus.sqlite --my-id agent_beta \
    --serve-forever --memory-db ./memory.sqlite

# Client (could be any process)
python3 agent_alpha.py --db ./bus.sqlite --my-id alpha_clientA \
    --target-id agent_beta --task compute_total \
    --intent "compute the total"

Pattern B — Worker pool with role-based fanout

# N replicas of the same role, atomic claim-locking handles distribution
python3 agent_beta.py --db ./bus.sqlite --my-id agent_beta_1 --serve-forever &
python3 agent_beta.py --db ./bus.sqlite --my-id agent_beta_2 --serve-forever &
python3 agent_beta.py --db ./bus.sqlite --my-id agent_beta_3 --serve-forever &

# Clients publish to role topic; one replica claims each
python3 agent_alpha.py --db ./bus.sqlite \
    --target-role worker --task compute_total \
    --intent "compute the total"

Pattern C — Reputation-weighted routing for high-stakes work

# Same pool, but Alpha filters by reputation
python3 agent_alpha.py --db ./bus.sqlite \
    --target-role worker --reputation-min 0.95 \
    --memory-db ./memory.sqlite \
    --task compute_total --intent "..."

§ 10 — BenefitsWhat this system does well

  1. Composition without coupling. Bus and verifier are independent; they compose because workers happen to include verification records in their replies.
  2. Trust travels as data. Every response includes claims, evidence, model, cost. Receivers apply their own policy.
  3. Workers self-check before publishing. Failures don't escape; remediation happens at the source.
  4. Independent failure modes. Cross-family verifier; separate processes; separate stores. Same-cause failures across layers are unlikely.
  5. Documentation flywheel. Gap reports → memory → system-prompt hints → tighter claims. Skills get sharper with every run.
  6. Bounded blast radius. TTL on every envelope. Hop count cap. Max-attempts cap. Per-attempt budget cap. Heartbeat staleness. Loops cannot persist.
  7. Reputation-weighted routing. Bad replicas get filtered out, with probation for new agents to prove themselves.
  8. Memory accumulation across runs. Gap reports dedup and reinforce — repeated observations matter more, not just more numerous.
  9. Three pluggable backends. Mock for offline, Anthropic for high stakes, Replicate for cost-sensitive volume. Swap with one CLI flag.
  10. Multi-hop trust composition. Weakest-link chain_status across all verification hops. Cost and gap reports aggregated end-to-end.
  11. Fully observable. Every message is a SQLite row. Every claim is recorded. Every conversation is replayable. Bus is the log.
  12. Production swap-in is one change per layer. SQLite→Postgres, mock→LLM, subprocess→real skill. Local changes only.

§ 11 — ImprovementsWhat still isn't done

Honest list of what's deferred. Some are v2 features; some are platform limitations; some are intentional non-goals.

Deferred for v2

  • Agentic verifier with tool access — current LLM verifier is single-call; behavioral claims that need execution are marked unverifiable
  • Human-in-the-loop attestation — high-stakes claims routed to a human channel for approval before publish
  • Cross-skill chain verification — verify that data from step 1 is what step 2 acted on
  • Streaming / incremental verification — currently end-of-skill only
  • Reputation decay — current scoring is raw success/total; old failures should weigh less than recent
  • Stake-weighted reputation — a failure on a $10k operation should matter more than one on a test query
  • Topic-claim reputation hooks — agents could self-deselect from role topics if their reputation drops

Platform-level (would change the substrate)

  • Postgres LISTEN/NOTIFY or Redis transport — for sub-100ms coordination
  • Cross-tenant isolation — multi-customer deployments need per-tenant buses or row-level ACLs
  • Rubric integration — verifications could feed Hyperagent's eval system as training data

Intentional non-goals

  • Recursive verification — verifier cannot itself call RunSkillVerified. Turtles-all-the-way-down is explicitly disallowed.
  • Schedule-based workflows — that's what Hyperagent's live-mode agents are for; the bus is for in-flight messaging.

§ 13 — GlossaryData shapes worth knowing

// Envelope — every message on the bus
{
  "id": "msg_<12 hex>",
  "conversation_id": "cnv_<12 hex>",
  "parent_conversation_id": "cnv_<12 hex>",    // for nested sub-conversations
  "from_agent": "agent_alpha",
  "to_agent": "agent_gamma",
  "topic": null,                              // or "role.worker"
  "msg_type": "request",                       // or "response" or "event"
  "reply_to": null,                           // for threading
  "hop_count": 0,                              // max 8
  "ttl_seconds": 300,
  "created_at": 1779140000.0,
  "payload": { /* application-defined */ }
}
// VerificationRecord — what every verifier returns
{
  "status": "verified",                       // or "failed" or "partial"
  "claims": [Claim],
  "gap_report": GapReport | null,
  "verifier_model": "meta/meta-llama-3-8b-instruct",
  "duration_ms": 8089,
  "cost_usd": 0.0002,
  "upstream_verification": VerificationRecord | null   // for chain composition
}
// ChainSummary — embedded in response payloads
{
  "chain_status": "verified",                  // weakest-link across all hops
  "hop_count": 2,
  "total_attempts": 4,
  "total_cost_usd": 0.0004,
  "total_duration_ms": 12340,
  "per_hop": [
    {"verifier_model": "...", "status": "...", ...},
    ...
  ],
  "merged_gap_report": GapReport | null             // deduped across all hops
}
// Claim — the atomic unit of verification
{
  "id": "claim_001",
  "type": "existential | structural | behavioral | factual | semantic | negative",
  "statement": "...",
  "evidence_required": "...",
  "evidence_collected": { /* what verifier observed */ } | null,
  "verdict": "pass | fail | unverifiable",
  "confidence": 0.0-1.0,
  "reasoning": "..."
}
// Memory — durable learning store row
{
  "id": "mem_<12 hex>",
  "scope": "skill.demo-compute-total",           // or "global" or "agent."
  "category": "gap",                          // or "failure_pattern", "reputation", "fact"
  "subject": "claim_004",
  "content": { /* category-specific */ },
  "importance": 0.7,                          // 0.0-1.0; bumps on reinforcement
  "use_count": 1,                              // increments each recall + reinforcement
  "created_at": 1779140000.0,
  "last_used_at": 1779143000.0
}
The data shapes are the contract. As long as you preserve these schemas, every other component composes. Add a 4th backend, a 4th agent role, a new memory category — the rest of the system doesn't change.
